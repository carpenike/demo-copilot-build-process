# Platform Incident Response Runbook

> Version: 1.0 | Owner: Platform Engineering | Last Updated: 2026-03

This runbook covers cross-cutting platform incidents that affect multiple services.
Service-specific runbooks are generated by @6-monitor and live at
`projects/<project>/operations/runbook.md`.

---

## Incident Severity Levels

| Severity | Definition | Response Time | Escalation |
|----------|------------|---------------|------------|
| **SEV-1** | Platform-wide outage, all services affected | 15 min | VP Engineering + on-call SRE |
| **SEV-2** | Multiple services degraded, partial outage | 30 min | On-call SRE + service owners |
| **SEV-3** | Single service degraded, workaround available | 1 hour | On-call SRE |
| **SEV-4** | Non-impacting anomaly, investigation needed | Next business day | Service owner |

## AKS Cluster Unhealthy

### Symptoms
- Multiple services returning 503
- `kube-apiserver` latency > 5s
- Node NotReady events across multiple nodes

### Diagnosis
```bash
# Check node status
kubectl get nodes -o wide

# Check system pod health
kubectl get pods -n kube-system

# Check recent events
kubectl get events --sort-by='.lastTimestamp' -A | tail -50

# Check API server latency
kubectl get --raw /healthz
```

### Remediation
1. Check Azure portal → AKS cluster → Diagnose and solve problems
2. If nodes are NotReady, check the VMSS instance health in Azure portal
3. If API server is unresponsive, check Azure Service Health for regional incidents
4. If etcd is degraded, do NOT restart — escalate to Azure Support (SEV-A)

### Escalation
- If not resolved within 30 minutes → page VP Engineering
- If Azure infrastructure issue → open Azure Support ticket (SEV-A)

## Azure Database for PostgreSQL — Connection Exhaustion

### Symptoms
- Services returning 500 with "too many connections" in logs
- Log Analytics query: `ContainerLog | where LogEntry contains "remaining connection slots are reserved"`
- Database CPU > 90% sustained

### Diagnosis
```sql
-- Check active connections by application
SELECT application_name, state, count(*)
FROM pg_stat_activity
GROUP BY application_name, state
ORDER BY count DESC;

-- Check for long-running queries
SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > interval '5 minutes'
  AND state != 'idle'
ORDER BY duration DESC;
```

### Remediation
1. Identify the service with the most connections (query above)
2. Check if the service recently deployed — connection pool misconfiguration?
3. Kill long-running queries if safe: `SELECT pg_terminate_backend(<pid>);`
4. If connection count is genuinely too low for traffic, scale up in Azure portal:
   - Flexible Server → Compute + storage → increase tier or max connections
5. Restart affected service pods to reset connection pools:
   ```bash
   kubectl rollout restart deployment/<service-name> -n <namespace>
   ```

### Escalation
- If queries cannot be killed safely → page database owner
- If Azure Flexible Server is unresponsive → Azure Support ticket

## Azure Cache for Redis — High Memory / Evictions

### Symptoms
- Alert: `RedisHighMemoryUsage` (memory > 80% of limit)
- Celery task failures with connection errors
- Increased latency on endpoints using cache

### Diagnosis
```bash
# Check Redis INFO from a pod with redis-cli
kubectl exec -it <pod> -n <namespace> -- redis-cli -h <redis-host> -a <password> INFO memory
kubectl exec -it <pod> -n <namespace> -- redis-cli -h <redis-host> -a <password> INFO stats
```

Key metrics to check:
- `used_memory_rss` vs `maxmemory`
- `evicted_keys` (should be 0 in normal operation)
- `connected_clients`

### Remediation
1. If evictions are high, identify large keys:
   ```bash
   redis-cli --bigkeys
   ```
2. Check if a service is storing unbounded data (missing TTL on keys)
3. Scale up Redis tier in Azure portal if memory is genuinely insufficient
4. If Celery queue is backed up, check worker replica count — scale up HPA

### Escalation
- If data loss from evictions → page service owner immediately

## CI/CD Pipeline Failures — Systemic

### Symptoms
- Multiple PRs failing CI at the same stage
- GitHub Actions runner queue times > 10 minutes
- Security scanning stage timing out

### Diagnosis
1. Check [GitHub Status](https://www.githubstatus.com/) for platform incidents
2. Check Azure Container Registry availability (ACR outage blocks `docker push`)
3. Check if a dependency update broke multiple projects:
   ```bash
   # Check recent Dependabot PRs
   gh pr list --label dependencies --state open
   ```

### Remediation
| Cause | Fix |
|-------|-----|
| GitHub Actions outage | Wait for resolution, monitor status page |
| ACR push failures | Check ACR health in Azure portal, verify network rules |
| Shared dependency break | Revert the dependency update, pin version |
| Runner queue backup | Check if self-hosted runners are healthy |
| Security scan timeout | Check CodeQL / Defender service health |

### Escalation
- If CI is blocked for > 1 hour across all projects → page Platform Engineering lead

## Secrets Rotation — Azure Key Vault

### When to Use
- Scheduled quarterly rotation
- After a suspected credential exposure
- When onboarding/offboarding a team member with direct Key Vault access

### Procedure
1. Generate new credential value (database password, API key, etc.)
2. Add the new version to Key Vault:
   ```bash
   az keyvault secret set --vault-name <vault> --name <secret-name> --value <new-value>
   ```
3. The External Secrets Operator will sync within its polling interval (default: 1 hour)
4. To force immediate sync:
   ```bash
   kubectl annotate externalsecret <name> -n <namespace> \
     force-sync=$(date +%s) --overwrite
   ```
5. Restart affected deployments to pick up the new secret:
   ```bash
   kubectl rollout restart deployment/<service-name> -n <namespace>
   ```
6. Verify the service is healthy:
   ```bash
   kubectl exec -it <pod> -n <namespace> -- curl -s localhost:8000/ready
   ```
7. Delete the old secret version from Key Vault after confirming all services are stable

### Post-Rotation Validation
- [ ] All services pass `/ready` health check
- [ ] No authentication errors in Log Analytics for the past 15 minutes
- [ ] Old secret version disabled in Key Vault
